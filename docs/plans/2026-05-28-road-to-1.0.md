**idêntico a `origin/main`** — 80 commits atrás de `fork/fixes`. Isso a deixou
obsoleta: a maior parte dos bugs do [issue #8](https://github.com/alkmimm/workspace-qdrant-mcp/issues/8)

reflete o estado de `fork/fixes` e só lista o que genuinamente resta.

Mais maduro do que a leitura sobre `main` sugeria. Os pontos operacionais que
separavam o projeto do 1.0 — normalização de path, starvation do queue, detecção de
projeto no Windows, SQLITE_BUSY — **já foram fechados em `fork/fixes`**. A amplitude

conjunto pequeno de itens de verificação, consistência e dogfooding — não capacidade.

## Trabalho que ainda resta para o 1.0

### 1. Fechar / verificar o issue #8 (P0)

O issue #8 ainda está **OPEN**, mas 3 dos 4 sub-itens já caíram. Falta:

- **Sub-item #4 — "destination failure on success path" (race de meio-sucesso):** não
  teve commit dedicado; o próprio issue prevê que a correção de #1 (feita via #9) faria
  a maioria sumir. **Ação:** confirmar num stack vivo (container) que os erros em
  `batch_processing.rs` (`qdrant_status`/`search_status = failed` no caminho de sucesso)
  não recorrem. Se recorrerem no tenant sobrevivente, investigar ordem de escrita do
  status por destino.
- **Ação de fechamento:** validar #1 (não há mais duplo tenant após #9) e #4 num stack
  limpo, então **fechar ou estreitar o issue #8**.

### 2. Guard de proto drift no CI (P1)

`#33` trouxe um proto fix pontual, mas **não** o check sistemático. O proto TS é um
subconjunto mantido à mão do proto Rust; RPCs faltando compilam e só estouram em runtime
("Unknown method").

- **Ação:** adicionar um check de CI que diffa a superfície de RPC do proto TS contra o
  `.proto` canônico do daemon e falha em RPC ausente/renomeado — ou gerar o cliente TS a
  partir do proto canônico.
- **Critério:** RPC faltando é pego no CI, não em runtime.

### 3. Reconciliar versão e docs (P1, barato)

- **Versão:** código em `0.1.3` (TS `package.json` + Cargo, confirmado em `main`),
  mas a tag `v0.4.1` continua solta (mais recente que `v0.1.3` no `creatordate`). O
  guard de consistência de versão já está no CI — **falta** cravar um esquema único e
  reconciliar/remover a tag `v0.4.1` órfã. *(Ainda aberto.)*
- **Docs:** ✅ `CLAUDE.md` em `main` já diz `v0.1.3` (não mais "v0.1.0"/"in
  development"). ✅ A contagem de MCP tools foi reconciliada para **10** em
  `CLAUDE.md` / `installation.md` / `08-api-reference.md` (`ef98a8b6e`). **Falta**
  varrer `README` e o restante de `docs/` pela mesma defasagem de versão/tools.
- **Critério:** tag mais recente, versões de crate e `package.json` concordam; nenhum doc
  chama de "in development" algo que já está pronto.

### 4. Robustez de runtime no hot path do daemon (P2)

Um panic no daemon derruba file watching, embedding e gRPC juntos. (Contagem de `unwrap`
fora de teste medida sobre main; **re-medir em fork/fixes**.)

- **Escopo deliberadamente estreito:** só o caminho crítico (queue processor, file
  processing, handlers gRPC, escrita de grafo). Não caçar todos os `unwrap()` do repo.
- **Critério:** sem `unwrap()`/`expect()` sobre valores falíveis (I/O, rede, parse, DB) no
  hot path; um panic em um item não aborta o loop de processamento.

### 5. Primeiros usuários / instalação limpa (P2 — maior salto de maturidade)

O projeto não tem usuários reais. Uma instalação limpa num ambiente novo
(Homebrew + Docker do zero) indexando um projeto real expõe gaps de onboarding melhor que
qualquer auditoria de código.

- **Ação:** documentar e verificar um caminho repetível "zero até a primeira busca".
- **Critério:** um usuário novo chega à primeira busca sem ler o código-fonte; gaps
  encontrados viram itens e são corrigidos.
